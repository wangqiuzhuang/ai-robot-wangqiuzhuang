# Week 13: 实验反思

## 1. 本周我学到了什么？

本周是课程中理论深度最高的一周，接触了强化学习在四足机器人上的应用。从零开始理解了三个核心概念：

**PPO 算法**：和之前接触的监督学习完全不同。监督学习需要标准答案（ground truth），而四足机器人的 12 个关节不可能人工标注最优动作。RL 只需要定义"什么是好"（走远加分、摔倒扣分），让策略自己探索。PPO 的关键创新是保守更新策略 —— 通过 clip 机制保证每次 Update 不会步子太大导致策略崩溃。

**Residual RL**：在基础步态控制器（正弦波）之上叠加 RL 修正。这个思路非常工程化 —— 不追求从零学走路的"纯粹 RL"，而是把已有的领域知识（正弦波生成步态）编码进基础控制器，RL 只负责处理偏差和特殊情况。收敛速度比纯 RL 快一个数量级。

**PyBullet 仿真**：轻量级的物理引擎，Python 原生接口，和 Gym 无缝对接。虽然没有 Gazebo 精确，但对于 RL 训练这种需要跑数千 episode 的场景，仿真速度比物理精度更重要。

## 2. 遇到了什么困难？怎么解决的？

最大的困难是理解 RL 的训练不稳定性。同样的超参数，跑三次训练得到三个完全不同的结果：一次学会了走、一次只会原地踏步、一次直接摆烂不动。这让我很困惑 —— 为什么同样的代码跑出不同结果？

排查后理解了：RL 训练对随机种子极度敏感。网络初始化、环境初始状态、动作探索的随机性叠加在一起，导致训练路径发散。解决思路是固定 seed 做对比实验，每次只改一个变量（比如只改 reward 函数、不改网络结构），才能确定"什么因素导致了什么结果"。

另一个困难是 reward engineering。最初的 reward 只包含"前进速度"，狗学会了用不自然的方式往前蹭（身体贴地、只用前腿划）。加了能耗惩罚和姿态惩罚后，步态才变得更像是"走路"而不是"挣扎"。这让我深刻体会到：**设计 reward 就是给 RL 定义"什么叫做好"，你定义得不准确，它就会钻空子。**

## 3. 这些知识和技能如何与更广泛的机器人概念关联？

强化学习在机器人领域的应用正在爆发。波士顿动力用 RL 让 Atlas 学会了跑酷，ETH Zurich 的 ANYmal 用 RL 在复杂地形上自主导航。虽然他们的规模比我这个课程实验大得多，但核心思路是一样的：

- **Sim-to-Real Transfer**：在仿真里训练（又快又安全），再把策略迁移到真实机器人。Domain Randomization（领域随机化）是迁移成功的关键 —— 在仿真里把环境参数随机化得足够多，真实环境的偏差就只是"其中一种变体"。
- **Reward 设计即行为设计**：想让机器人做什么样的行为，就要设计什么样的 reward。这是一个需要直觉和经验的活。
- **RL + 传统控制的结合**：Residual RL 的思路很有启发性 —— 把成熟的 PID/MPC 作为基础，RL 只做高级决策和异常处理。这种分层架构在工业界比"纯 RL 端到端"更可靠。

接下来我想做的：把 W13 的 RL 模型和 W12 的 WebSocket 遥控链路结合起来 —— 网页上按"自动探索"按钮 → 触发 RL 策略接管机器狗控制 → 策略输出通过 WebSocket 发给仿真环境 → 实时状态回传网页显示。如果做成了，这就是一个完整的"AI 自动控制 + Web 远程监控"系统，可以直接作为 W14 的进阶特性。
