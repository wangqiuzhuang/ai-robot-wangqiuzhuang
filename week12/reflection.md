# Week 12: 实验反思

## 1. 本周我学到了什么？

本周是课程中技术栈最复杂的一周，打通了「手机摄像头 → Tailscale 组网 → WSL Python/OpenCV → ArUco 识别」这样一条跨设备的实时视频处理管道。涉及的技术包括：Tailscale 虚拟组网、SSH 远程连接、WebSocket 实时通信、OpenCV 视频解码、ArUco 增强现实标记识别。

最深的体会是：**把这些独立的技术组件串成一个工作流，比单独学习每项技术难得多。** Tailscale 安装很顺利，但手机浏览器调用摄像头的 HTTPS 要求就折腾了一阵；WebSocket 传输单帧没问题，但连续视频流的稳定性和延迟又是另一回事；ArUco 单独测能识别，但在手机传来的压缩 JPEG 上就是不稳定。工程里 80% 的时间花在模块之间的接口和调试上，而不是模块本身的实现。

另一个重要收获是对 ArUco 标记的理解。之前只觉得它是个"奇怪的二维码"，现在明白了它的核心价值是 **6-DoF 位姿估计** —— 这是二维码完全做不到的。单个 ArUco 标记加上 PnP 算法就能告诉机器人"自己相对于标记的位置和朝向"，这对于室内定位、AR 标定、机械臂视觉伺服都至关重要。

## 2. 遇到了什么困难？怎么解决的？

**困难一**：浏览器刷新一直超时。原因是在 WSL 里跑的程序端口挂起，新的请求发到旧端口上。解决方法：`kill` 掉旧进程后重新 `python run`。

**困难二**：ArUco 在手机传来的 JPEG 压缩帧上识别率不稳定。排查后发现是 JPEG 压缩率的问题 —— 手机端为了降低带宽把 quality 压到 50，导致标记的黑色边框变模糊。解决方案：把 quality 调到 90，虽然每帧体积大了约 40%，但识别率从 ~70% 提升到 ~95%。

**困难三**：理解 WebSocket 和 HTTP 的区别花了一些时间。以前只知道"HTTP 是网页用的"，不明白为什么传视频必须用 WebSocket。看了 MDN 的文档和对比数据后才理解：30fps 的视频如果用 HTTP POST 每帧一个请求，光是 HTTP 头的开销每秒就 ~15KB，还伴随大量 TCP 握手/挥手。WebSocket 一次握手后帧头仅 2-6 bytes，延迟从 ~200ms 降到 ~10ms。

## 3. 这些知识和技能如何与更广泛的机器人概念关联？

本周的实验是在搭建一个**机器人远程视觉系统的最小原型**。把这个原型放大：

- **Tailscale 组网 → ROS2 DDS 网络**：多机器人场景下，每个机器人跑自己的 ROS2 节点，通过 DDS 自动发现和数据共享。Tailscale mesh 拓扑和 ROS2 的 peer-to-peer 架构在理念上是一致的。
- **WebSocket 传输 → ROS2 话题**：WebSocket 的双向实时通信，本质上就是 ROS2 的 publish/subscribe 模型的 web 化实现。ROS2 也有 `rosbridge_suite` 把 ROS2 话题暴露成 WebSocket，让网页可以直接和机器人通信。
- **ArUco 识别 → 机器人定位**：ArUco 本身是 Camera-based Localization（基于相机的定位）的一种实现。在真实的机器人系统中，定位可能靠 LiDAR SLAM、UWB、视觉里程计，但 ArUco 在受控环境（实验室、工厂、展会）里是性价比最高的方案。

如果要把本周的链路产品化，我会做的改进：在手机端加一个"画质-延迟"的动态平衡策略（WiFi 好时传高清、4G 时压低码率）；Python 端加上帧率统计和带宽监控；ArUco 检测失败时自动降级到光流法跟踪。
