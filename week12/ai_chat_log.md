# Week 12: AI 对话记录

> 以下为本周实验中与 AI 助手的真实对话摘录，共 6 轮。主要围绕 Tailscale 跨设备组网、WebSocket 视频流传输及 ArUco 标记识别。

---

## 第 1 轮

**我：**
Tailscale 和普通 VPN 有什么区别？为什么它能穿透 NAT 让我手机直接访问 WSL 里的服务？

**AI：**
Tailscale 本质上是基于 **WireGuard** 的 mesh VPN，但它有几个关键创新：

1. **NAT 穿透（STUN + DERP）**：Tailscale 会用 STUN 协议探测 NAT 类型，尝试 UDP 打洞。如果打洞失败，自动走 DERP 中继服务器，保证总能连通。你手机（移动网络）和 WSL（校园网 NAT 之后）能直连，就是靠这套机制。

2. **Mesh 拓扑**：不是星型（所有流量走中心服务器），而是点对点直连。你的手机和 WSL 之间数据不需要经过 Tailscale 服务器。

3. **MagicDNS**：每台设备分配 `100.x.y.z` 的虚拟 IP 和 `<hostname>.tailxxxxx.ts.net` 域名，不需要记 IP。

普通 VPN 需要你在服务器端开端口、配防火墙，Tailscale 全自动。

## 第 2 轮

**我：**
手机浏览器访问 `http://<wsl-ip>:5000` 调用摄像头，为什么必须是 HTTPS 才能用摄像头？HTTP 不行吗？

**AI：**
这是浏览器的安全策略。`getUserMedia()` API（调用摄像头/麦克风）要求页面必须通过 HTTPS 加载（或者 `localhost` 例外）。原因是：摄像头是极度敏感的隐私设备，如果 HTTP 明文网页就能调用摄像头，中间人可以劫持页面偷偷录像。

但你的场景有点绕：WSL 的 Python 服务端没有 SSL 证书，所以你用 HTTP 跑在 WSL 本地，手机通过 Tailscale 访问 WSL 的 IP。这种情况下：
- 如果 WSL 的 Python 直接 serve HTTP → 手机浏览器拒绝摄像头权限
- 解决方法：用 ngrok 做 HTTPS 隧道，或者生成自签名证书（手机要手动信任，很麻烦）

课程代码应该已经处理了这个问题（可能是用 Flask 的 adhoc SSL 或 ngrok）。

## 第 3 轮

**我：**
WebSocket 和普通的 HTTP 请求有什么区别？为什么传视频流必须用 WebSocket 而不是一直发 HTTP POST？

**AI：**

| | HTTP | WebSocket |
|---|---|---|
| 连接方式 | 请求-响应，一次一连接 | 持久连接，双向通信 |
| 开销 | 每次请求都有 HTTP 头（~500-800 bytes） | 初始握手后帧头仅 2-6 bytes |
| 方向 | 客户端主动请求 | 双向，服务器可主动推送 |
| 适用 | API 调用、网页加载 | 实时数据流、聊天、游戏 |

传视频流用 HTTP POST 的话，每帧一张 JPEG 就要建一次 TCP 连接 + HTTP 握手，30fps 的摄像头就是每秒 30 次握手，延迟和带宽开销都不可接受。

WebSocket 建好连接后，JPEG 帧直接通过 WebSocket 二进制帧发送，没有额外的 HTTP 头，延迟极低。你代码里的流程是：手机浏览器 `getUserMedia()` 拿视频帧 → canvas 截图转 JPEG → WebSocket 发到 WSL → Python OpenCV 解码显示。

## 第 4 轮

**我：**
ArUco 标记的原理是什么？它和二维码有啥区别？为什么机器人定位用 ArUco 而不是二维码？

**AI：**
ArUco（Augmented Reality University of Cordoba）是一种**基准标记**，专为 AR 和机器人定位设计。

和二维码的区别：

| | 二维码 | ArUco |
|---|---|---|
| 编码内容 | URL/文本（高密度） | 仅一个数字 ID（低密度） |
| 识别距离 | 近（需要看清细节） | 远（黑白对比强） |
| 姿态估计 | ❌ 不支持 | ✅ 可计算 6-DoF 位姿 |
| 实时性 | 慢（需要纠错解码） | 快（主要是图像处理） |
| 应用 | 支付、信息分享 | 机器人定位、AR 标定 |

ArUco 标记的核心设计：黑色边框 + 内部二值化矩阵。识别流程是：
1. 自适应阈值提取轮廓
2. 多边形近似过滤（找矩形）
3. 透视变换矫正
4. 内部网格二值化读取 ID
5. 已知标记尺寸 → PnP 算法解算相机位姿

机器人定位之所以用 ArUco，就是因为它能告诉你"相对于标记的 6 自由度位姿"（x, y, z, roll, pitch, yaw）。

## 第 5 轮

**我：**
我这周搭的手机 → Tailscale → WSL → OpenCV → ArUco 这条链路，在实际机器人上能怎么用？

**AI：**
这条链路本质上是一个**远程视觉感知管道**，可以衍生出很多实际应用：

1. **远程机器人遥控 + 视觉反馈**：把 OpenCV 处理结果回传到手机网页（比如在画面上叠加 ArUco 检测框），你能在手机上看到机器人"看到的"，然后遥控。

2. **移动端 AI 推理前置**：手机摄像头拍到的画面，手机端先跑一个轻量级模型（比如人脸检测），然后把检测结果（而不是原始视频流）发给 WSL/机器人，节省带宽。

3. **AR 定位**：在环境中贴 ArUco 标记，机器人摄像头拍到就能定位自己。你搭的这条链路就是机器人定位系统的一个简化版。

4. **多机协同**：多台手机（每台绑在一个小机器人上）通过 Tailscale mesh 连接到中心控制节点（WSL），每个机器人传回自己的视频流，中心节点统一做多目标跟踪。

## 第 6 轮

**我：**
我在实际测试中发现 ArUco 标记有时候识别不到，主要是什么因素影响？怎么优化？

**AI：**
影响 ArUco 识别率的主要因素：

1. **光照**：标记上的阴影或不均匀光照会导致二值化失败。解决：用自适应阈值（`cv2.aruco.DetectorParameters` 里的 `adaptiveThreshWinSizeMin`）或加补光灯。

2. **标记尺寸**：在摄像头画面中太小（比如标记距离太远，画面占比 <5%）就识别不到。粗略公式：标记在画面中最少要占 20×20 像素才能可靠识别。

3. **运动模糊**：摄像头或标记在移动时，快门不够快导致模糊。解决：降低移动速度、提高环境光、或降低摄像头曝光时间。

4. **遮挡**：标记边框被部分遮挡 → 轮廓检测失败。解决：多放几个标记，用标记板（ArUco Board）。

5. **摄像头分辨率**：太低（<640×480）的时候远处的标记细节丢失。你手机摄像头分辨率肯定够了，关键是传输时有没有被压缩。

你在手机 → WebSocket → Python 这条链路上，如果 JPEG 压缩率太高，标记的边框会变模糊，影响识别。可以试试降低 JPEG 压缩率（提高 quality 参数到 90+）。
